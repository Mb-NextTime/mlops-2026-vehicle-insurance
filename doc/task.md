# Постановка задачи и ML-дизайн

## 1. Описание данных и бизнес-задача
* **Источник данных:** Набор данных «Vehicle Insurance Data» (Kaggle).
* **Суть задачи:** Мультиклассовая классификация. Прогнозирование типа транспортного средства (`TYPE_VEHICLE`), содержащего 11 уникальных классов, на основе параметров страхователя и полиса.
* **Эмуляция потоковости:** В качестве временной оси (timestamp) используется колонка `INSR_BEGIN` (Дата начала действия страховки). Данные сортируются хронологически, после чего подаются в систему "батчами" (по умолчанию 5000 строк). Для демонстрации устойчивости пайплайна в новые батчи искусственно внедряется 5% случайных пропусков.

## 2. Сбор и Анализ данных (Data Quality)
При поступлении нового батча система выполняет следующие шаги:
1. **Расчет метапараметров**: Для числовых признаков вычисляется математическое ожидание и дисперсия, подсчитывается общее число строк и пропусков. Данные сохраняются в JSON.
2. **Анализ качества (DQ)**: 
   * Рассчитывается доля пропусков в каждой колонке.
   * Ищутся аномалии (outliers) в числовых данных с использованием межквартильного размаха (IQR).
3. **Ассоциативные правила**: К категориальным признакам (с кардинальностью <10) применяется One-Hot бинаризация, после чего алгоритм **Apriori** ищет логические связи (с минимальным *support=0.1* и *confidence=0.5*).
4. **Очистка**: Удаляются признаки-дубликаты и колонки, где процент пропусков превышает допустимый порог (настраивается в конфиге, по умолчанию 30%).

## 3. Проектирование модели (ML Pipeline)

### Препроцессинг (Feature Engineering)
Признаковое пространство обрабатывается с помощью `sklearn.pipeline.Pipeline`:
* **Числовые признаки**: Пропуски заполняются медианой (`SimpleImputer`), после чего признаки масштабируются (`StandardScaler`).
* **Категориальные признаки**: Пропуски заполняются константой 'missing'. Применяется `OneHotEncoder` (с флагом `handle_unknown='ignore'` для обработки новых категорий в будущих батчах).
* **Заморозка**: Объект `ColumnTransformer` (препроцессор) обучается (`fit`) исключительно на первом батче и сохраняется. На последующих батчах применяется только `transform`, что гарантирует неизменность размерности входа для моделей.

### Выбранные алгоритмы и гиперпараметры
В конвейере параллельно обучаются 2 модели. Обе поддерживают **инкрементальное обучение** (Incremental Learning), что позволяет не переобучать их с нуля на каждом батче.

1. **Random Forest Classifier (Случайный лес)**
   * *Гиперпараметры*: Базовое число деревьев `n_estimators=50`, глубина не ограничена (до чистоты листа).
   * *Механика дообучения*: Включен параметр `warm_start=True`. При поступлении нового батча к ансамблю добавляется 10 новых деревьев, обученных на свежих данных.
   * *Обработка Target Drift*: Реализован механизм гарантии наличия всех 11 классов в таргете (dummy-rows injection), чтобы предотвратить ошибку размерности при усреднении предсказаний леса.

2. **MLP Classifier (Многослойный перцептрон)**
   * *Гиперпараметры*: 2 скрытых слоя `(64, 32)`, функция активации ReLU, оптимизатор Adam.
   * *Механика дообучения*: Использование метода `partial_fit()`, который обновляет веса нейросети на основе градиентов нового батча без потери предыдущей памяти.

## 4. Валидация и Хранение артефактов (Model Registry)
* **Hold-out валидация**: Каждый батч перед препроцессингом делится на *train* и *test* (по умолчанию 80/20).
* **Метрики**: Рассчитываются `Accuracy` и `F1-macro` (учитывая мультиклассовость задачи).
* **Model Registry**: Все версии моделей сохраняются в директорию `storage/models/` (например, `model_rf_batch_1.pkl`). Состояние реестра отслеживается в файле `model_registry.json`.
* **Champion Model**: При валидации каждого батча система сравнивает точность текущих моделей с историческим максимумом. Лучшая модель помечается в реестре как чемпионская и автоматически используется при вызове сценария `inference` для скоринга новых внешних данных.

## 5. Архитектура CI/CD (GitHub Actions)

В качестве платформы для автоматизации конвейера выбран **GitHub Actions**.

**Выбранный сценарий: CRON-инкрементальное обучение**
1. **Триггеры:** Конвейер реагирует на `push`/`PR`, а также выполняется по таймеру (CRON) раз в сутки.
2. **Тестирование:** Перед запуском обучения выполняется проверка целостности среды утилитой `pytest` (проверка конфигов и наличия данных).
3. **Инкрементальность между запусками:** Поскольку GitHub Actions запускает изолированные виртуальные машины (runners), для сохранения "памяти" системы между разными cron-запусками реализован механизм кэширования (`actions/cache`). Папка `storage/` кэшируется в конце каждого прогона и восстанавливается в начале следующего. Это позволяет сборщику данных продолжать чтение файла с нужной строки, а ML-моделям — дообучаться, а не инициализироваться с нуля.
4. **Визуализация (Dashboard):** Сформированный файл `summary_report.md` динамически инжектируется в переменную `$GITHUB_STEP_SUMMARY`. Это позволяет анализировать падение/рост метрик (Accuracy/F1) прямо в веб-интерфейсе GitHub, не скачивая логи вручную.
